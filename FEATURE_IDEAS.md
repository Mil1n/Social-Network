## 💬 Чаты и сообщения

- **Статусы сообщений**: отправлено, доставлено, прочитано.
- **Typing indicator**: «Алекс печатает…».
- **Ответы на сообщения**: reply preview внутри bubble.
- **Закреплённые сообщения**: важная информация сверху чата.
- **Поиск по истории чата**: поиск текста по текущему диалогу.
- **Реакции**: 👍 ❤️ 😂 🔥 👀 для сообщений и постов.
- **Пересылка сообщений**: отправка сообщения в другой чат или канал.
- **Черновики**: сохранять недописанный текст в каждом чате.
- **Голосовые сообщения**: запись аудио в браузере через MediaRecorder API.

## 📣 Каналы и социальная лента

- **Комментарии к постам**: обсуждения под публикациями.
- **Отложенные публикации**: планирование постов на дату и время.
- **Черновики постов**: сохранение незавершённых публикаций.
- **Публичные и приватные каналы**: разные уровни доступа.
- **Пригласительные ссылки**: `join/:inviteCode` для групп и каналов.
- **Закреплённые посты**: важные публикации вверху канала.
- **Статистика канала**: просмотры, подписчики, реакции, популярные посты.

## 👤 Профили и социальный граф

- **Фотография профиля** и обложка.
- **Статус пользователя**: online, offline, last seen.
- **Контакты/друзья**: запросы, подтверждения, список контактов.
- **Блокировка пользователей**: запрет сообщений и просмотра профиля.
- **Настройки приватности**: кто видит bio, online, каналы и контакты.
- **Публичная страница профиля**: короткая ссылка на пользователя.

## 🛡️ Модерация и безопасность

| Идея | Что даёт |
| --- | --- |
| Очередь жалоб с изменением статуса | Модераторы смогут закрывать, отклонять и эскалировать жалобы |
| Бан/разбан пользователей | Базовое управление нарушителями |
| Мут в чатах | Временное ограничение активности |
| Audit log фильтры | Быстрый поиск действий администраторов |
| Rate limiting | Защита от спама и brute force |
| bcrypt/Argon2 | Безопасное хранение паролей |
| Session expiration | Сессии не будут жить бесконечно |
| CSRF/XSS hardening | Безопаснее для публичного запуска |

## ⚡ Realtime

Текущий MVP работает через обычные REST-запросы. Следующий большой шаг — realtime.

Рекомендуемый путь:

1. Добавить **Server-Sent Events** для уведомлений и новых сообщений.
2. Затем перейти на **WebSocket**, когда появятся typing indicators, presence и звонки.
3. Для масштабирования вынести события в **Redis Pub/Sub** или очередь сообщений.

Что появится после realtime:

- новые сообщения без обновления страницы;
- live-уведомления;
- online/offline статусы;
- typing indicator;
- быстрые реакции;
- live-модерация.

## 🗄️ Переход на production-хранилище

Сейчас данные хранятся в JSON-файле. Для реального продукта лучше перейти на:

- **PostgreSQL** — пользователи, чаты, сообщения, каналы, роли, жалобы;
- **Redis** — кеш, rate limiting, online presence, pub/sub;
- **S3-compatible storage** — изображения, видео, документы, голосовые сообщения;
- **Meilisearch/OpenSearch** — быстрый поиск по сообщениям, постам и пользователям.

## 💰 Монетизация

Этичные варианты, которые не ломают доверие:

- premium-профили;
- расширенное хранилище файлов;
- платные приватные каналы;
- донаты авторам;
- корпоративные workspace-тарифы;

Лучше избегать агрессивной рекламы и продажи пользовательских данных — для личной социальной сети доверие важнее краткосрочной выручки.

## 🔥 Новые идеи после Telegram Lite update

### 1. Настоящий realtime без перезагрузки

**Почему это важно:** социальная сеть начинает ощущаться живой только тогда, когда сообщения, реакции и уведомления появляются мгновенно.

Минимальный план:

1. Добавить `GET /api/events` на Server-Sent Events.
2. Отправлять события `message:new`, `message:edited`, `message:deleted`, `reaction:changed`, `notification:new`.
3. На frontend подписаться через `EventSource`.
4. Автоматически обновлять открытый чат и счётчики без ручного клика.

### 2. Нормальная система контактов

Сейчас пользователь вручную копирует ID другого пользователя. Это удобно для MVP, но плохо для реального продукта.

Что добавить:

- запросы в контакты;
- принятие/отклонение заявок;
- список контактов;
- быстрый старт личного чата из профиля;
- блокировку пользователя;
- приватность: кто может писать первым.

### 3. Личные диалоги 1-на-1

Сейчас чаты создаются как группы по ID участников. Следующий шаг — отдельный UX для личных сообщений:

- кнопка «Написать» в карточке пользователя;
- автоматическое создание или открытие существующего direct chat;
- отдельная иконка/тип `direct`;
- запрет дублирующихся личных диалогов с одним пользователем.

### 4. Файлы и медиа

Сейчас вложение — это URL. Для настоящего мессенджера нужно хранение файлов.

Минимальный MVP:

- `POST /api/files` для загрузки файла;
- локальная папка `uploads/` для dev-режима;
- ограничения размера и MIME-type;
- превью картинок в сообщениях;
- скачивание документов;
- очистка файлов при удалении сообщения.

Production-вариант:

- S3-compatible storage;
- thumbnails worker;
- virus scanning;
- signed URLs;
- quotas per user/workspace.

### 5. Темная тема и настройки интерфейса

Быстрый UI-win, который сразу делает приложение «дороже» визуально.

- dark/light/system theme;
- выбор accent color;
- compact/comfortable режим;
- сохранение настроек в профиле;
- переключатель в правой панели профиля.

### 6. Роли и права в группах/каналах

Для социальной сети нужны не только `admin` и `user`, а гибкая модель прав.

Возможные роли:

- owner;
- admin;
- moderator;
- member;
- muted;
- banned.

Права:

- отправлять сообщения;
- удалять чужие сообщения;
- публиковать в канал;
- приглашать участников;
- менять настройки;
- просматривать audit log.

### 7. Улучшенная модерация

Текущие жалобы уже сохраняются, но их нужно превратить в рабочий процесс.

- статусы жалоб: `open`, `reviewing`, `resolved`, `rejected`;
- комментарий модератора;
- кнопки «удалить контент», «забанить», «замьютить»;
- фильтры по статусу, типу объекта и дате;
- отдельная страница moderation queue.

### 8. Поиск как в Telegram

Поиск должен стать глобальным:

- пользователи;
- чаты;
- каналы;
- сообщения;
- посты;
- файлы.

Для MVP можно начать с простого поиска по JSON/PostgreSQL. Для production лучше добавить Meilisearch или OpenSearch.

### 9. Онбординг и демо-данные

Чтобы проект сразу производил впечатление:

- seed-команда `npm run seed`;
- демо-пользователи;
- демо-чаты;
- демо-каналы;
- приветственный бот/канал;
- подсказки в пустых состояниях.

### 10. Переход к production backend

Самый важный технический этап после UX-доработок:

- PostgreSQL вместо JSON-файла;
- миграции;
- repository/service слои;
- bcrypt/Argon2;
- secure session cookies;
- rate limiting;
- structured logs;
- integration tests для API.

## 🎯 Самый полезный следующий шаг

Я бы начал с **контактов + личных диалогов 1-на-1**. Это даст пользователю главный социальный сценарий: найти человека → добавить в контакты → написать ему без ручного копирования ID. После этого уже логично добавлять realtime, файлы и полноценную модерацию.
